# Security Review — 2026-07-11

Full-project security review of ProfitSync (`profitsync.net`), commissioned to
"make sure the project is properly secure." Conducted as a multi-agent fan-out
audit across nine security dimensions, with **every** candidate finding put through
an independent adversarial-refutation pass before being accepted, then the confirmed
issues fixed on branch `fix/security-hardening`.

## Scope & method

- **Surfaces audited:** authorization / IDOR across all 110 `api/_routes/**` handlers,
  auth-guard coverage, platform-admin RBAC/capabilities, SQL injection (every `sql`` /
  raw-SQL sink), secrets / crypto / webhook signature verification, the native
  (Capacitor Android) OAuth + deep-link + network surface, file-upload & SSR/XSS,
  quota bypass & mass-assignment, and cross-tenant data exposure incl. the new tags code.
- **Method:** 9 read-only finder agents (one per dimension) → each finding independently
  verified by an adversarial reviewer prompted to *refute* it → only findings that
  survived with ≥7/10 confidence were accepted. 13 agents, 0 errors, ~1.04M tokens.
- **Hand-verification:** the money/quota path finding was re-derived by hand from the
  actual code before any fix (per the project's "never ship a ledger change you only
  reasoned about" rule) — which is what caught that the finder's recommended fix would
  have *introduced* a worse bypass (see below).

## Findings fixed

### H-1 (HIGH → fixed) — `is_system` mass-assignment + quota inconsistency

**Where:** `api/_routes/transactions.ts`, `api/_routes/transactions/group.ts`,
`api/_routes/wealth/accounts.ts`, `api/_routes/wealth/accounts/[id].ts`,
`api/_lib/quota.ts`, `api/_routes/wealth/transfer.ts`.

**What:** Two coupled problems around the `transactions.is_system` flag.
1. **Mass assignment:** the transaction POST and split-group POST read `is_system`
   straight from the request body (`isSystem: !!is_system`). It's server-state, never
   something a client should set. It has no filtering effect *today*, but it's a latent
   footgun: the moment any future code treats system rows specially (e.g. hides them from
   a total), a user could set the flag to manipulate it.
2. **Quota inconsistency (the reported HIGH):** wealth "system" transactions
   (opening-balance on account create, and balance-adjustment on PATCH) were inserted
   **without** a quota check, yet `checkTransactionQuota` **counted** them. On the free
   plan (30 tx/client) this both (a) let those rows accrue past the intended limit and
   (b) let a user exhaust their own per-client quota via repeated balance adjustments,
   locking themselves out of adding real transactions.

**Why the naive fix was wrong:** the finder recommended "add a quota check to the wealth
system-tx path" *or* "exclude system rows from the count." Excluding-from-count **alone**
would have been a *worse* bug — because `is_system` was body-settable, any user could then
POST unlimited `is_system:true` transactions and bypass the quota entirely. Adding a check
to the wealth path instead risked a ledger-integrity gap (account `currentBalance` set but
its backing opening-balance row skipped when at quota).

**Fix (consistent with the existing Space-transfer exemption in `transfer.ts:55`):**
- Made `is_system` **server-only** — removed it from both request-body handlers, so
  user-created rows are always `isSystem = false`.
- Exempted server-set `isSystem = true` rows (internal, off-P&L ledger entries) from the
  per-client quota count in `checkTransactionQuota` **and** the inline transfer count.

Net effect: user transactions still count and are still gated; internal wealth ledger
rows neither consume nor bypass the quota; and the flag can no longer be attacker-set.

### L-1 (LOW → fixed) — Service-token length/timing side-channel

**Where:** `api/_lib/auth.ts` `requireServiceToken`.

**What:** the constant-time compare was guarded by `a.length === b.length && timingSafeEqual(...)`.
The `&&` short-circuits on a length mismatch, so response timing leaked whether a supplied
token matched the secret's length. Practical exploitability is very low (network jitter
dwarfs the signal; the tokens are 64-char hex, so length adds ~no search-space reduction),
hence LOW — but it's a trivial, clean fix.

**Fix:** compare **SHA-256 digests** instead of raw buffers. Digests are always 32 bytes,
so `timingSafeEqual` never sees a length mismatch and there's no short-circuit — the
comparison is uniform for every candidate and every input length.

## Defense-in-depth hardenings (on dismissed items)

These candidates were **verified NOT exploitable**, but tightened anyway since the cost was one line:

- **D-1 — org-scope on the `transactions/[id]` GET enrichment re-fetch.** The re-fetch
  ran `where(id)` only. It was safe (the row is already ownership-checked above and
  `client_id` is immutable), but the query now re-asserts `clients.organization_id = orgId`
  so it self-defends regardless of future PATCH changes.
- **D-2 — HTTPS assertion for `VITE_API_BASE_URL`.** Android 9+ blocks cleartext by
  default (so the "MITM the native app" path was not exploitable), but `api-base.ts` now
  throws at boot if the base URL is set to non-`https` (localhost excepted), so a
  misconfigured build fails loudly instead of shipping bearer tokens over http.

## Findings dismissed (verified false positives)

- **TOCTOU IDOR in `transactions/[id]` GET** — *not a vuln*. Transactions are org-scoped via
  the immutable `client_id` FK; the PATCH handler never allows `client_id` to change, so no
  check-to-fetch race window exists. (Hardened anyway — see D-1.)
- **Cleartext `VITE_API_BASE_URL` MITM** — *not exploitable*. Requires both a compromised
  build env AND an explicit cleartext network-security-config; the OS blocks cleartext by
  default on the app's target SDK. (Hardened anyway — see D-2.)

## What was confirmed solid (no action needed)

- **Auth & org-scoping** — every route resolves the active org from `x-org-id` via a
  membership check; queries scope by `orgId`, never `userId` alone. The org-auth cache is
  keyed by `(userId, headerOrgId)` so it can't cross tenants.
- **Route-guard coverage** — all handlers gate on `requireAuth` / `requireAdminCap` /
  `requireServiceToken`; the CI route-guard sweep enforces it.
- **Admin RBAC** — capability-based, super-admin-exclusive caps enforced server-side, custom
  roles sanitized to the grantable set on read *and* write.
- **SQL injection** — Drizzle parameterizes interpolations; the single raw-SQL call
  (in `flow.ts`) interpolates only a whitelisted `bucket` literal.
- **Webhook** — Dodo signature verified (HMAC-SHA256) before any mutation.
- **SSR/XSS** — blog markdown is sanitized via a `sanitize-html` allowlist; no unsafe raw
  HTML injection on untrusted input.
